## 项目概述

**elomon**（代号：cardmon / 宝可梦战棋）是一个用 Godot 4.6 开发的宝可梦风格战棋 RPG 原型。战斗在 20×20 网格上进行，采用 CTB（Charge Time Battle）系统——每个单位根据自身速度持续积累行动力（AP），达到 100 时轮到该单位行动。

## 运行游戏

用 Godot 4.6 编辑器打开项目，按 F5（运行项目）。主场景为 `battle/Battle.tscn`。当前项目视口为 `640×360`，20×20 网格会超出首屏，后续通过相机滚动查看完整战场。

`addons/godot_mcp` 是编辑器自动化插件，与游戏逻辑无关。

## 核心系统一览

| 系统 | 入口文件 | 职责 |
|------|----------|------|
| 战斗循环 | `battle/Battle.gd` | 协调所有子系统，驱动回合状态机、同步率与后备能力提取 |
| 资源事件 | `battle/Battle.gd` | 统一记录 AP/同步率增减与行动节奏事件，供临时反馈、日志和未来正式 UI 复用 |
| 属性倍率 | `core/TypeChart.gd` | 管理属性克制表，按单位属性列表连乘伤害倍率 |
| 状态类型 | `core/StatusTypes.gd` | 统一管理状态 id、正负面、分类、驱散、触发时机、持续时机、叠加规则、短标签和说明文案 |
| 状态实例 | `core/StatusInstance.gd` / `core/StatusResolver.gd` | 表示单位身上的通用状态，并按行动开始/结束、使用技能、受伤前后、移动后等钩子触发 |
| CTB 计时 | `battle/ctb/CTBSystem.gd` | 每帧累加 AP，AP 满时暂停并通知 |
| CTB 显示 | `battle/ctb/CTBBar.gd` | AP 轴/顺序轴双视图，用单条横轴展示单位头像位置和未来行动顺序；头像悬停显示 AP、速度和预计行动位次 |
| 网格 | `grid/Grid.gd` | 管理单位位置、移动/攻击范围计算、点击事件、敌方威胁覆盖层 |
| 单位运行时 | `units/Unit.gd` | HP、AP、格子位置、移动压制、临时状态条等动态状态 |
| 单位静态数据 | `units/data/UnitData.gd` | 属性、定位文案、技能列表、AI 战术偏好与蓄力攻击配置（`.tres` 实例） |
| 技能静态数据 | `skills/SkillData.gd` | 伤害/治疗、射程、定位文案、效果列表与范围半径 |
| 技能效果数据 | `skills/SkillEffectData.gd` / `skills/SkillEffectResolver.gd` | 描述并结算技能附加效果，例如添加状态、目标行动条变化，后续扩展中毒、降防、加攻等 |
| 敌方 AI | `units/UnitAI.gd` | 无状态静态类，按 `AIProfile` 权重评分目标；支持猎手、克制、守卫、压制和范围等战术偏好 |
| 行动菜单 | `ui/ActionMenu.gd` | 底部指令栏，显示当前行动单位、真实技能名、卡牌消耗、提取形态、回收和待机/结束状态；悬停说明以小字显示在按钮下方 |
| 战斗日志 | `battle/Battle.gd` | 结构化记录已结算行动，并在右下 HUD 临时滚动展示 |
| 战斗结算 | `battle/Battle.gd` | 胜负后显示击败数、同步率操作、训练师状态和剩余同步率 |
| 全局枚举/常量 | `core/Enums.gd` | `UnitType`、`ElementType`、`AIProfile`、`BattleState`、`ActionState`、网格常量 |

同步率当前没有硬上限。自然回复只补到 100，训练师攻击、宝可梦攻击等主动收益可以继续超过 100；同步率暂不跨战斗继承。

训练师初始同步率为 100。每个训练师回合最多结算 1 次同步率操作：指令卡、提取和召唤互斥；移动、技能和回收不占用这次机会。训练师菜单中同步率操作统一收在「同步率」顶层菜单下，再展开「指令 / 提取 / 召唤」。

额外召唤使用递增同步率费用：第 1 次 100，第 2 次 160，第 3 次 220。召唤次数按本场战斗已成功额外召唤次数累计，回收不会降低下一次召唤费用。回收返还 50 同步率，回收后的宝可梦保留当前 HP；再次召唤仍按递增费用支付，且不会满血重置。

战斗开始前会显示轻量战前准备面板：选择样板战、2 只开局宝可梦和训练师默认提取形态。第一战默认配置为火狐兽 + 藤藤兽上场，训练师默认提取水跃兽；第二样板战默认配置为藤藤兽 + 电花鼠上场，训练师默认提取水跃兽。第二样板战当前定位为“侧翼压制测试”：中线由地属性守卫和 Boss 牵制，上下翼由长射程水系、飞系猎手和草系控制制造站位压力。战斗中再次提取消耗 100 同步率。

战前准备确认后会先显示一次“第一场战斗”提示面板，说明胜利目标、属性 2 倍克制、同步率操作限制、指令卡类型和敌方范围按钮；玩家点“开始行动”后 CTB 才开始推进。

战斗 UI 采用 v0 边缘布局：顶部低矮状态条承载 CTB、同步率和快捷按钮；行动菜单固定为底部指令栏，左侧显示当前行动单位，下方小字显示悬停说明；行动日志默认折叠到右侧抽屉，伤害预览临时占用同一区域。棋盘单位悬停会显示轻量小卡，包含阵营、属性、HP/AP、速度、移动、技能和临时状态。临时 HUD、行动菜单、CTB 条、伤害预览、战前提示和结算面板都使用半透明深色背景和细边框，避免直接裸字叠在战场上导致可读性下降。

提取形态在 UI 中统一按“宝可梦 / 属性 / 定位 / 提取技能”展示。新增后备能力时，`EXTRACT_DEFS` 需要同时维护 `element_label`、`role`、`skill_name`，供战前准备、HUD、悬停说明和日志复用。

当前同步形态会占用对应后备宝可梦：例如训练师正在提取水跃兽时，水跃兽仍显示在后备中但标记为“同步”，不能被召唤；切换到其他提取形态后才释放。

当前玩家精灵的内容基线见 `.claude/unit_skill_roster.md`。火狐兽偏火系爆发/范围，藤藤兽偏草系远程/地系快控，水跃兽偏支援/治疗，电花鼠偏雷系高速/飞系快招，冰羽兽偏压制/重控。单个技能仍按单属性结算，但同一只精灵可以携带不同属性技能；技能说明只描述机制事实，不直接写成攻略提示。

临时状态的基础定义统一由 `StatusTypes.gd` 描述，包含状态/印记分类、是否可驱散、触发时机、持续时机、叠加规则和短标签。当前持续类型包含持续到消耗、下次行动、下次攻击、下次移动、下次受伤、释放者下次行动前和固定触发次数。单位脚下会显示小状态条，悬停说明和卡牌说明也应复用这些持续类型。移动压制使用旧轻量字段兼容；新接入的防御变化使用 `StatusInstance`。

当前 MVP 暂不做战斗内捕捉。稳定度字段仍作为后续设计预留，但 UI、伤害预览、日志和胜负判定不读取稳定度。

技能的基础伤害/治疗仍保留在 `SkillData` 上；技能附加效果统一使用 `SkillData.effects` 中的 `SkillEffectData`。当前已接入 `ADD_STATUS`（例如移动压制、防御变化）和 `AP_DELTA`（例如目标行动条-20%）。防御变化会作为 `StatusInstance` 挂到目标身上，在下次受伤前修改防御并消耗；`move_penalty`、`target_ap_delay` 只作为旧字段兼容，新增技能不要继续堆独立字段。

战斗结束后会显示结算面板。结算只统计真实落地的事件：击败数来自单位死亡，同步率操作来自正式战斗日志，不统计预览、取消或未提交的 pending 日志。
