# 关键约定

## 坐标系
格子坐标为 `Vector2i(col, row)`（x=列，y=行）。世界像素中心 = `grid_pos * CELL_SIZE + CELL_SIZE/2`。

内部数组索引为 `_grid[row][col]`，即 `_grid[y][x]`。

当前网格大小为 20×20，`CELL_SIZE=32`。项目视口为 `640×360`，网格总高度大于视口高度，后续用相机滚动解决显示范围问题。

## AP 经济
单位行动结束后不将 AP 归零，而是扣除本回合行动条内部结算值，溢出部分保留；若重招扣到负数，则需要先回补负 AP 再攒到 `MAX_AP (100)`。速度快的单位因此长期保持先手优势。

MVP 版本中 AP 是行动条内部资源。移动不消耗 AP，也不做普通冲刺；额外移动只通过训练师刷高速组件获得。没使用技能的行动内部扣除 50 AP，玩家侧提示为“下次行动提前50%”。大部分技能为标准行动，内部扣除 100 AP，不显示成本提示；只有少量快招/重招通过 `SkillData.ap_cost` 改变行动节奏，例如 80 显示“下次行动提前20%”，120 显示“下次行动推后20%”。AP 可以被消耗成负数，为以后过载机制预留空间。

AP 和同步率的变化必须走统一资源事件：`resource_event_emitted(event)`。事件至少需要包含 `event_type`、`resource_type`、`amount`、`before`、`after`、`reason`、`actor`、`target` 和 `metadata`。当前临时 UI 可以只显示简短浮字，但正式 UI、音效、战报、教程提示都应优先消费这套事件，而不是从 HUD 文本或日志文本反推资源变化。

同步率没有硬上限。自然回复使用 `SYNC_NATURAL_CAP (100)` 作为软上限，只能把同步率补到 100；训练师攻击、宝可梦攻击等主动行为可以继续把同步率推到 100 以上。同步率暂不跨战斗继承。

训练师初始同步率为 100。战斗开始前会通过轻量准备面板选择 2 只开局宝可梦和 1 个默认提取形态，默认提取不消耗同步率。每个训练师回合最多结算 1 次同步率操作，由 `_turn_has_support_action` 表示；指令卡、战斗中提取、召唤和回收都占用这次机会。移动和技能不占用同步率操作。进入目标选择后按 `Esc` 取消不消耗机会；可撤销的提取若按 `Esc` 撤销，需要返还同步率并恢复 `_turn_has_support_action=false`。

战前准备确认后，`Battle.gd` 需要先显示一次“第一场战斗”提示面板，说明胜利目标、属性 2 倍克制、同步率每回合 1 次、指令卡类型和敌方范围按钮。提示面板关闭前不要启动 `CTBSystem`，避免玩家读提示时战斗已经推进。

`CTBSystem` 暂停时不推进任何单位 AP。恢复跑条后必须先检查已经满 AP 的单位并触发 `unit_ready`，避免外部效果或初始化状态造成满 AP 单位被跳过。

`CTBBar` 需要支持两种展示：速度跑条视图和行动轴视图。行动轴视图按当前 AP 与速度预测未来行动顺序，速度快的单位可以重复出现；第一版使用紧凑短名显示，避免遮挡同步率 HUD。速度跑条视图需要标出 `READY` 和 `NEXT`，行动轴视图需要区分当前行动单位与下一位行动单位。

## 移动与技能
每个单位每次行动最多移动一次，固定移动距离取 `UnitData.move_range`，由 `has_moved` 限制。高速组件会给目标宝可梦下一次移动增加临时距离，移动完成后清除该加成。

每个单位每次行动最多使用 1 个技能，由 `has_acted` 限制。MVP 中非训练师技能确认结算后会自动结束该单位回合；训练师技能结算后仍可继续使用指令卡、召唤、回收或手动结束行动。任何单位使用技能后都不能再移动；未来如果要支持攻击后移动，应通过明确的道具、状态或特性开启，而不是作为默认规则。技能的 `ap_cost` 字段是行动条内部结算值，只在非标准时转译为“下次行动提前/推后 X%”给玩家。

## `has_acted` 标志
每回合开始时重置为 `false`，使用技能后设为 `true`，防止同一回合内二次施法。

## 场景节点名称
`Battle.gd` 通过 `@onready` 路径引用子节点：

```
$Grid → GridManager（脚本文件为 grid/Grid.gd）
$CTBSystem → CTBSystem
$UI/CTBBar → CTBBar
$UI/ActionMenu → ActionMenu
$UI/ResultLabel → Label
```

这些名称必须与 `Battle.tscn` 中的节点名完全一致，否则运行时报错。

## 代码风格

始终使用 Godot 4.x 最新语法，避免沿用 Godot 3 的写法（如 `connect()` 的旧签名、`yield`、`setget` 等）。

信号连接统一使用以下形式，不使用字符串方式的 `connect("signal_name", ...)`：

```gdscript
# 正确
node.signal_name.connect(callback)

# 错误
node.connect("signal_name", self, "callback")
```

## 资源文件
单位和技能数据是 Godot `.tres` 文件，请通过 Godot 编辑器的 Inspector 修改，不要手动编辑其文本格式。

MVP 样板战为了快速迭代，暂时在 `Battle.gd` 中动态创建 `UnitData` 和 `SkillData`。稳定后再把数据迁回 `.tres` 资源。

单位属性使用 `UnitData.element_types` 作为结算来源，`element_type` 只作为主属性兼容字段。新增或切换单位属性时优先调用 `set_element_types()`，不要只改 `element_type`。属性倍率统一走 `TypeChart.gd`，避免在战斗脚本里写死克制判断。当前单属性克制为 2 倍；若未来添加双属性，倍率按防御属性列表连乘。

## MVP 输入约定
训练师行动菜单包含指令卡、后备能力提取、后备召唤和回收按钮。只有训练师回合可以使用这些操作；其他己方宝可梦只能移动、使用技能或等待。行动菜单需要根据当前单位上下文收缩：非训练师回合隐藏所有训练师指挥按钮；训练师回合先显示「同步率」顶层按钮，展开后用「指令 / 提取 / 召唤」分组切换，避免长菜单一次性铺满。

当前指令卡需要先点按钮，再点目标格。目标合法性由 `Battle.gd` 检查，并通过顶部提示文字反馈。

后备能力提取不是指令卡，不需要选择目标格。提取使用 `_reserve_units` 判断可用性：只有对应宝可梦仍在后备中时才能提取。战斗中提取消耗 `EXTRACT_COST (100)` 同步率并设置 `_trainer_extract_id`，随后训练师的 `UnitData.element_types` 和 `skills` 被替换为对应后备形态；效果持续到下一次提取。当前同步形态会占用对应后备宝可梦：它仍显示在后备中，但召唤按钮需要标记为同步占用，点击时不能召唤；切换到其他提取形态后才释放。若对应宝可梦被召唤上场，按钮需要显示离场状态并在悬停说明中提示不可用原因。提取形态的可读信息由 `EXTRACT_DEFS` 统一维护，必须包含 `element_label`、`role`、`skill_name`，UI 统一按“宝可梦 / 属性 / 定位 / 提取技能”展示。

训练师在本回合刚提取能力、且还没有移动、使用技能、刷卡、召唤或回收时，按 `Esc` 需要撤销这次提取：恢复提取前的属性、技能和形态，并返还同步率。任何实际影响战局的操作发生后，都要清除这次提取撤销机会。

后备召唤由 `SUMMON_DEFS` 驱动，按钮与说明需要根据 `_reserve_units` 显示可用或离场状态。新增可召唤后备时，需要同时补 `POKEMON_IDS`、`SUMMON_DEFS`、`EXTRACT_DEFS`、菜单按钮和 MVP 文档说明；战前准备标签从 `EXTRACT_DEFS` 的展示字段自动生成。

选择移动、技能、指令卡或召唤后，可以按 `Esc`（Godot 默认 `ui_cancel`）取消当前选择并回到行动菜单。技能伤害/治疗预览确认中按 `Esc` 会返回技能选格状态，而不是直接退出整个行动。

己方单位移动后，只要本回合还没有使用技能且没有使用卡牌/召唤/回收，按 `Esc` 可以撤回到本回合开始位置。

我方回合空闲时点击敌方单位，会显示该单位的移动范围和按最远技能计算的攻击威胁。选择我方移动时，也会显示该单位移动后按最远技能计算的攻击覆盖范围。蓝色表示可移动格，红色表示攻击/威胁范围。

战斗 UI 提供“敌方范围”按钮，可在任何时候切换敌方全体威胁覆盖层。该覆盖层显示敌方移动后按最远技能计算的攻击范围，并由 `Grid.gd` 作为 threat cells 维护；普通移动/技能高亮可以临时覆盖它，清除高亮后需要恢复 threat cells。

行动菜单需要显示真实技能名，而不是只显示“技能1/技能2”。鼠标停在技能、卡牌、提取、召唤或回收按钮上时，顶部提示区需要显示射程、范围、费用、冷却、属性变化、持续条件或消耗时机等关键信息。

技能必须先进入预览，再确认结算。攻击预览需要显示目标 HP 变化、预计伤害、属性克制/抵抗和总伤害；治疗预览需要显示目标 HP 回复量。单体和范围技能共用同一套预览面板；范围技能额外在地图上标出受影响单位的预览数字。非训练师确认技能后自动结束回合；训练师确认技能后停留在行动菜单，用“结束”按钮结束当前行动。

当前 MVP 暂不做战斗内捕捉，也不展示稳定度。单位和技能数据中保留稳定度字段作为后续设计预留，但 UI、伤害预览、日志和胜负判定不读取这些字段。

指令卡当前包括高速组件、小型护盾、火力插件、弱点标记、战术换位和属性校准。弱点标记只能选择敌方目标，目标下次受到伤害提高 50%，受击后消耗；战术换位让训练师与目标己方宝可梦交换格子；属性校准只能选择己方宝可梦，使其下一次伤害技能使用训练师当前提取属性结算克制，攻击后消耗。火力插件和属性校准可以叠加，伤害预览和日志必须展示这些修正。

临时状态类型统一由 `core/StatusTypes.gd` 管理，不要在 UI 文案里各自发明持续描述。当前持续类型包括：持续到消耗或移除、下次行动有效、下次攻击有效、下次移动有效、下次受伤有效、释放者下次行动前有效。单位运行时仍保留轻量字段（护盾、下次攻击强化、弱点标记、属性校准、额外移动、蓄力预警），但状态条、悬停说明和后续正式 buff 系统都应从 `StatusTypes` 读取 id、短标签、颜色和持续文本。

动作预览需要保留结构化数据：`_current_action_preview` / `action_preview_updated(preview)` 包含 actor、技能名、目标格、AP before/after/cost、同步率 before/after/delta 和目标预估。临时 UI 可以只显示 AP 前后变化，但正式预览、行动轴动画和教学提示都应复用这份 preview 数据。

战斗日志只记录已经结算并影响战局的行动：移动、技能、指令卡、召唤、回收、敌方行动和待机/结束。选择按钮、预览范围、伤害预览、非法目标提示以及 Esc 取消不写入正式日志。移动和训练师提取能力属于可撤销操作时，先记录到 `_pending_turn_logs`；若按 Esc 撤销，直接丢弃 pending 日志；若发生技能、卡牌、召唤、回收或结束行动，再提交到正式展示日志。

日志要同时服务玩家可读性和后续调试/复盘：展示层可以简洁，但内部 record 需要保留事件类型、actor/target 阵营、位置、HP/AP、伤害倍率、同步率消耗/获得等细节。单位稳定 id 当前尚未设计，先在日志 metadata 中保留 `unit_id` TODO 占位。展示文本中属性克制/抵抗倍率放在伤害数字前，避免玩家误读为伤害后的补充说明。行动日志用颜色区分单位名：我方蓝色、敌方红色、中立/野生柔和黄色。

战斗结算面板只统计已落地事件：击败数由敌方单位 `died` 信号累计，同步率操作数从正式 `_battle_logs` 中的 `card`、`extract`、`summon`、`recall` 事件统计。预览、Esc 取消和未提交的 `_pending_turn_logs` 不进入结算。胜负判定需要等当前技能或 AI 行动日志写入后再弹结算，避免最后一击的日志和同步率奖励晚于结算页。

训练师倒下后不会立刻失败，但训练师相关操作视为离线：不能再使用指令卡、召唤或回收。胜负判断改为己方全部单位倒下才失败。

同步率 HUD 必须常驻显示当前值；每次实际获得同步率时，需要显示短暂的获得量反馈，避免资源变化无感。

同步率 HUD 还需要常驻显示主要获得规则、自然回复软上限、当前训练师形态和当前后备名单，避免玩家只能靠文档理解资源来源与提取能力来源。

训练师行动菜单需要把指令卡、提取和召唤统一收在「同步率」顶层菜单下，再展开「指令」「提取」「召唤」三个子菜单。同步率操作已使用后，顶层菜单显示「同步率已用」并禁用，避免玩家误以为还能继续结算同步率操作。

行动菜单需要在 `show_at()` 中夹在视口范围内，并通过内部 `ScrollContainer` 对长菜单限高滚动，避免在 640x360 视口下溢出。新增按钮时要注意按钮高度和菜单总高；新增训练师操作时优先放进现有分组，而不是直接追加到所有菜单状态里。选择指令卡或回收时，只高亮合法目标；若范围内没有合法目标，应留在菜单并提示原因，不能进入空目标选择状态。

## MVP 视觉约定
单位占位色块优先表达阵营，而不是元素属性：我方为柔和蓝色，敌方为柔和红色，中立/野生为柔和黄色。

单位脚下显示简易 HP 条；受到实际 HP 伤害时，会出现短暂的红色伤害数字。护盾完全吸收伤害时只更新标签和血条，不显示伤害数字。

单位脚下同时显示小状态条，用 1-2 字短标签表达关键临时状态，例如护盾、强、弱、校水、移、蓄。状态颜色按 `StatusTypes.Polarity` 区分：增益偏蓝，减益偏红，中性预警偏黄。这个状态条是临时信息密度方案，未来有正式 UI 后仍应保留同一套状态类型数据。

## 地形约定
地形数据由 `Grid.gd` 的 `_terrain[y][x]` 维护。高亮清除时必须恢复地形颜色，而不是统一恢复普通地面颜色。

当前 MVP 只启用普通地面和草地显示，不启用燃烧地面、点燃草地或地形伤害。若后续恢复地形伤害，需要重新检查移动撤回规则。

## 敌方预警
厚血大怪的蓄力攻击使用 `Grid.gd` 的 warning cells 显示橙色预警格。普通移动/攻击范围高亮可以临时覆盖预警色，但清除高亮后必须恢复预警色。大怪死亡或蓄力结算时需要清除预警格。
