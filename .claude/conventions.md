# 关键约定

## 坐标系
格子坐标为 `Vector2i(col, row)`（x=列，y=行）。世界像素中心 = `grid_pos * CELL_SIZE + CELL_SIZE/2`。

内部数组索引为 `_grid[row][col]`，即 `_grid[y][x]`。

当前网格大小为 20×20，`CELL_SIZE=32`。项目视口为 `640×360`，网格总高度大于视口高度，后续用相机滚动解决显示范围问题。

## AP 经济
单位行动结束后不将 AP 归零，而是扣除本回合行动条内部结算值，溢出部分保留；若重招扣到负数，则需要先回补负 AP 再攒到 `MAX_AP (100)`。速度快的单位因此长期保持先手优势。

MVP 版本中 AP 是行动条内部资源。移动不消耗 AP，也不做普通冲刺；额外移动只通过训练师刷高速组件获得。没使用技能的行动内部扣除 50 AP，玩家侧提示为“下次行动提前50%”。大部分技能为标准行动，内部扣除 100 AP，不显示成本提示；只有少量快招/重招通过 `SkillData.ap_cost` 改变释放者自己的行动节奏，例如 80 显示“自身下次行动提前20%”，120 显示“自身下次行动推后20%”。AP 可以被消耗成负数，为以后过载机制预留空间。

AP 和同步率的变化必须走统一资源事件：`resource_event_emitted(event)`。事件至少需要包含 `event_type`、`resource_type`、`amount`、`before`、`after`、`reason`、`actor`、`target` 和 `metadata`。当前临时 UI 可以只显示简短浮字，但正式 UI、音效、战报、教程提示都应优先消费这套事件，而不是从 HUD 文本或日志文本反推资源变化。

同步率没有硬上限。自然回复使用 `SYNC_NATURAL_CAP (100)` 作为软上限，只能把同步率补到 100；训练师攻击、宝可梦攻击等主动行为可以继续把同步率推到 100 以上。同步率暂不跨战斗继承。

训练师初始同步率为 100。战斗开始前会通过轻量准备面板选择 2 只开局宝可梦和 1 个默认提取形态，默认提取不消耗同步率。每个训练师回合最多结算 1 次同步率操作，由 `_turn_has_support_action` 表示；指令卡、战斗中提取和召唤占用这次机会。移动、技能和回收不占用同步率操作。进入目标选择后按 `Esc` 取消不消耗机会；可撤销的提取若按 `Esc` 撤销，需要返还同步率并恢复 `_turn_has_support_action=false`。

战前准备确认后，`Battle.gd` 需要先显示一次“第一场战斗”提示面板，说明胜利目标、属性 2 倍克制、同步率每回合 1 次、指令卡类型和敌方范围按钮。提示面板关闭前不要启动 `CTBSystem`，避免玩家读提示时战斗已经推进。

`CTBSystem` 暂停时不推进任何单位 AP。恢复跑条后必须先检查已经满 AP 的单位并触发 `unit_ready`，避免外部效果或初始化状态造成满 AP 单位被跳过。

`CTBBar` 需要支持两种横轴展示：`AP轴` 和 `顺序`。AP 轴按当前 AP 把所有存活单位头像放在同一条从左到右推进的轴上，最右侧代表 AP 满；顺序轴按预测行动顺序把头像固定排在同一条轴上，下一位行动者在最右侧，速度快的单位可以重复出现。MVP 阶段用单位名首字代替头像，并用阵营色区分敌我。头像需要响应悬停，展示单位名、AP、速度、当前状态和预计行动位次。

## 移动与技能
每个单位每次行动最多移动一次，基础移动距离取 `UnitData.move_range`，由 `has_moved` 限制。高速组件会给目标宝可梦下一次移动增加临时距离，移动完成后清除该加成；移动压制会降低目标下次行动的可移动距离，并在行动结束后清除。

每个单位每次行动最多使用 1 个技能，由 `has_acted` 限制。MVP 中非训练师技能确认结算后会自动结束该单位回合；训练师技能结算后仍可继续使用指令卡、召唤、回收或手动结束行动。己方宝可梦可以在自己的行动中直接回收自身，并立刻结束行动。任何单位使用技能后都不能再移动；未来如果要支持攻击后移动，应通过明确的道具、状态或特性开启，而不是作为默认规则。技能的 `ap_cost` 字段是释放者行动条内部结算值，只在非标准时转译为“自身下次行动提前/推后 X%”给玩家；命中目标的行动条控制必须使用 `SkillEffectData.EffectType.AP_DELTA`，不要混用 `ap_cost`。

## `has_acted` 标志
每回合开始时重置为 `false`，使用技能后设为 `true`，防止同一回合内二次施法。

## 场景节点名称
`Battle.gd` 通过 `@onready` 路径引用子节点：

```
$Grid → GridManager（脚本文件为 grid/Grid.gd）
$CTBSystem → CTBSystem
$UI/CTBBar → CTBBar
$UI/ActionMenu → ActionMenu
$UI/ResultLabel → Label
```

这些名称必须与 `Battle.tscn` 中的节点名完全一致，否则运行时报错。

## 代码风格

始终使用 Godot 4.x 最新语法，避免沿用 Godot 3 的写法（如 `connect()` 的旧签名、`yield`、`setget` 等）。

信号连接统一使用以下形式，不使用字符串方式的 `connect("signal_name", ...)`：

```gdscript
# 正确
node.signal_name.connect(callback)

# 错误
node.connect("signal_name", self, "callback")
```

## 资源文件
单位和技能数据是 Godot `.tres` 文件，请通过 Godot 编辑器的 Inspector 修改，不要手动编辑其文本格式。

MVP 样板战为了快速迭代，暂时在 `Battle.gd` 中动态创建 `UnitData`、`SkillData` 和 `SkillEffectData`。稳定后再把数据迁回 `.tres` 资源。第一战与第二样板战目前通过战前准备下拉框切换；第二样板战的少量专用技能和上下翼站位也先保留在 `_spawn_level2_enemies()` 附近。新增临时样板战时优先复用现有生成函数，等关卡数量变多后再抽正式关卡数据表。

单位属性使用 `UnitData.element_types` 作为结算来源，`element_type` 只作为主属性兼容字段。新增或切换单位属性时优先调用 `set_element_types()`，不要只改 `element_type`。属性倍率统一走 `TypeChart.gd`，避免在战斗脚本里写死克制判断。当前单属性克制为 2 倍；若未来添加双属性，倍率按防御属性列表连乘。

## MVP 输入约定
训练师行动菜单包含指令卡、后备能力提取、后备召唤和回收按钮。指令卡、提取和召唤只有训练师回合可以使用；己方宝可梦自己的回合也可以使用回收。行动菜单需要根据当前单位上下文收缩：非训练师回合隐藏所有训练师指挥按钮，只保留基础动作和自回收；训练师回合先显示「同步率」顶层按钮，展开后用「指令 / 提取 / 召唤」分组切换，避免长菜单一次性铺满，回收作为独立按钮常驻。

当前指令卡需要先点按钮，再点目标格。目标合法性由 `Battle.gd` 检查，并通过顶部提示文字反馈。

后备能力提取不是指令卡，不需要选择目标格。提取使用 `_reserve_units` 判断可用性：只有对应宝可梦仍在后备中时才能提取。战斗中提取消耗 `EXTRACT_COST (100)` 同步率并设置 `_trainer_extract_id`，随后训练师的 `UnitData.element_types` 和 `skills` 被替换为对应后备形态；效果持续到下一次提取。当前同步形态会占用对应后备宝可梦：它仍显示在后备中，但召唤按钮需要标记为同步占用，点击时不能召唤；切换到其他提取形态后才释放。若对应宝可梦被召唤上场，按钮需要显示离场状态并在悬停说明中提示不可用原因。提取形态的可读信息由 `EXTRACT_DEFS` 统一维护，必须包含 `element_label`、`role`、`skill_name`，UI 统一按“宝可梦 / 属性 / 定位 / 提取技能”展示。

训练师在本回合刚提取能力、且还没有移动、使用技能、刷卡、召唤或回收时，按 `Esc` 需要撤销这次提取：恢复提取前的属性、技能和形态，并返还同步率。任何实际影响战局的操作发生后，都要清除这次提取撤销机会。

后备召唤由 `SUMMON_DEFS` 驱动，按钮与说明需要根据 `_reserve_units` 显示可用或离场状态。额外召唤费用由 `_summon_count` 计算：第 1 次 100，第 2 次 160，第 3 次 220；召唤成功后计数递增，回收不会降低下一次召唤费用。回收返还 50 同步率，把宝可梦放回 `_reserve_units`，并用 `_reserve_hp_by_name` 保存当前 HP；再次召唤时恢复该 HP，但 AP 使用召唤入场默认值。新增可召唤后备时，需要同时补 `POKEMON_IDS`、`SUMMON_DEFS`、`EXTRACT_DEFS`、菜单按钮和 MVP 文档说明；战前准备标签从 `EXTRACT_DEFS` 的展示字段自动生成。

选择移动、技能、指令卡或召唤后，可以按 `Esc`（Godot 默认 `ui_cancel`）取消当前选择并回到行动菜单。技能伤害/治疗预览确认中按 `Esc` 会返回技能选格状态，而不是直接退出整个行动。

己方单位移动后，只要本回合还没有使用技能、同步率操作或回收，按 `Esc` 可以撤回到本回合开始位置。

我方回合空闲时点击敌方单位，会显示该单位的移动范围和按最远技能计算的攻击威胁。选择我方移动时，也会显示该单位移动后按最远技能计算的攻击覆盖范围。蓝色表示可移动格，红色表示攻击/威胁范围。

战斗 UI 提供“敌方范围”按钮，可在任何时候切换敌方全体威胁覆盖层。该覆盖层显示敌方移动后按最远技能计算的攻击范围，并由 `Grid.gd` 作为 threat cells 维护；普通移动/技能高亮可以临时覆盖它，清除高亮后需要恢复 threat cells。

行动菜单需要显示真实技能名，而不是只显示“技能1/技能2”。鼠标停在技能、卡牌、提取、召唤或回收按钮上时，底部指令栏的小字说明需要显示射程、范围、费用、冷却、属性变化、持续条件或消耗时机等关键信息。

技能必须先进入预览，再确认结算。攻击预览需要显示目标 HP 变化、预计伤害、属性克制/抵抗和总伤害；治疗预览需要显示目标 HP 回复量。单体和范围技能共用同一套预览面板；范围技能额外在地图上标出受影响单位的预览数字。非训练师确认技能后自动结束回合；训练师确认技能后停留在行动菜单，用“结束”按钮结束当前行动。

当前 MVP 暂不做战斗内捕捉，也不展示稳定度。单位和技能数据中保留稳定度字段作为后续设计预留，但 UI、伤害预览、日志和胜负判定不读取这些字段。

指令卡当前包括高速组件、小型护盾、火力插件、弱点标记、战术换位和属性校准。弱点标记只能选择敌方目标，目标下次受到伤害提高 50%，受击后消耗；战术换位让训练师与目标己方宝可梦交换格子；属性校准只能选择己方宝可梦，使其下一次伤害技能使用训练师当前提取属性结算克制，攻击后消耗。火力插件和属性校准可以叠加，伤害预览和日志必须展示这些修正。

临时状态类型统一由 `core/StatusTypes.gd` 管理，不要在 UI 文案里各自发明持续描述。当前持续类型包括：持续到消耗或移除、下次行动有效、下次攻击有效、下次移动有效、下次受伤有效、释放者下次行动前有效。单位运行时仍保留轻量字段（护盾、下次攻击强化、弱点标记、属性校准、额外移动、移动压制、蓄力预警），但状态条、悬停说明和后续正式 buff 系统都应从 `StatusTypes` 读取 id、短标签、颜色和持续文本。

动作预览需要保留结构化数据：`_current_action_preview` / `action_preview_updated(preview)` 包含 actor、技能名、目标格、AP before/after/cost、同步率 before/after/delta 和目标预估。临时 UI 可以只显示 AP 前后变化，但正式预览、行动轴动画和教学提示都应复用这份 preview 数据。

战斗日志只记录已经结算并影响战局的行动：移动、技能、指令卡、召唤、回收、敌方行动和待机/结束。选择按钮、预览范围、伤害预览、非法目标提示以及 Esc 取消不写入正式日志。移动和训练师提取能力属于可撤销操作时，先记录到 `_pending_turn_logs`；若按 Esc 撤销，直接丢弃 pending 日志；若发生技能、卡牌、召唤、回收或结束行动，再提交到正式展示日志。回收日志需要记录返还同步率和保留 HP。

日志要同时服务玩家可读性和后续调试/复盘：展示层可以简洁，但内部 record 需要保留事件类型、actor/target 阵营、位置、HP/AP、伤害倍率、同步率消耗/获得等细节。单位稳定 id 当前尚未设计，先在日志 metadata 中保留 `unit_id` TODO 占位。展示文本中属性克制/抵抗倍率放在伤害数字前，避免玩家误读为伤害后的补充说明。行动日志用颜色区分单位名：我方蓝色、敌方红色、中立/野生柔和黄色。

战斗结算面板只统计已落地事件：击败数由敌方单位 `died` 信号累计，同步率操作数从正式 `_battle_logs` 中的 `card`、`extract`、`summon`、`recall` 事件统计。预览、Esc 取消和未提交的 `_pending_turn_logs` 不进入结算。胜负判定需要等当前技能或 AI 行动日志写入后再弹结算，避免最后一击的日志和同步率返还晚于结算页。

训练师倒下后不会立刻失败，但训练师相关操作视为离线：不能再使用指令卡、召唤或训练师回收。场上宝可梦自己的自回收仍按普通行动处理；胜负判断改为己方全部单位倒下才失败。

同步率 HUD 必须常驻显示当前值；每次实际获得同步率时，需要显示短暂的获得量反馈，避免资源变化无感。

同步率 HUD 还需要常驻显示主要获得规则、自然回复软上限、当前训练师形态和当前后备名单，避免玩家只能靠文档理解资源来源与提取能力来源。

训练师行动菜单需要把指令卡、提取和召唤统一收在「同步率」顶层菜单下，再展开「指令」「提取」「召唤」三个子菜单。同步率操作已使用后，顶层菜单显示「同步率已用」并禁用，避免玩家误以为还能继续结算同步率操作。回收不在同步率菜单内，不能被「同步率已用」禁用。

行动菜单需要固定为底部指令栏，而不是跟随单位漂浮。`show_at()` 保留旧参数兼容，但实际按当前视口把菜单贴到底部；指令栏左侧显示“轮到 X”，中间显示按钮，下方小字显示当前提示或鼠标悬停说明。内部使用可换行按钮容器和 `ScrollContainer` 限高滚动，避免在 640x360 视口下溢出。新增按钮时要注意按钮高度和菜单总高；新增训练师操作时优先放进现有分组，而不是直接追加到所有菜单状态里。选择指令卡或回收时，只高亮合法目标；若范围内没有合法目标，应留在菜单并提示原因，不能进入空目标选择状态。

## MVP 视觉约定
单位占位色块优先表达阵营，而不是元素属性：我方为柔和蓝色，敌方为柔和红色，中立/野生为柔和黄色。

单位脚下显示简易 HP 条；受到实际 HP 伤害时，会出现短暂的红色伤害数字。护盾完全吸收伤害时只更新标签和血条，不显示伤害数字。

单位脚下同时显示小状态条，用 1-2 字短标签表达关键临时状态，例如护盾、强、弱、校水、移、蓄。状态颜色按 `StatusTypes.Polarity` 区分：增益偏蓝，减益偏红，中性预警偏黄。这个状态条是临时信息密度方案，未来有正式 UI 后仍应保留同一套状态类型数据。

技能附加效果统一写在 `SkillData.effects` 中，单个效果使用 `SkillEffectData` 描述。当前支持 `ADD_STATUS` 和 `AP_DELTA`，后续中毒、降防、加攻、AP回复变化等都应扩展 `SkillEffectData` / `StatusTypes` / `SkillEffectResolver`，不要继续在 `SkillData` 上追加 `poison_turns`、`defense_down`、`attack_up` 这类一次性字段。`move_penalty` 和 `target_ap_delay` 仅为旧 `.tres` 或过渡代码保留兼容。

移动压制属于技能型减益，短标签为“缚”。它由 `SkillEffectData.EffectType.ADD_STATUS + StatusTypes.StatusId.MOVE_PENALTY` 产生，降低目标下次行动的移动距离，并在该目标行动结束后清除。不要把移动压制写成永久减速或跳过行动；MVP 只验证“站位被轻度干扰”这件事。

行动条压制属于技能命中效果，由 `SkillEffectData.EffectType.AP_DELTA` 产生，命中存活目标后直接改变目标当前 AP；例如 `value=-20` 表示目标行动条后退 20%。技能说明和日志必须写“目标行动条-20%”，避免和释放者自己的“自身下次行动推后20%”混淆。

异常状态与数值 Buff/Debuff 先通过 `StatusTypes.StatusId` 预留类型：`POISON`、`BURN`、`ATTACK_MOD`、`DEFENSE_MOD`、`AP_REGEN_MOD`。当前 MVP 尚未结算这些状态的持续逻辑；真正接入时需要统一补：预览、实际结算、行动开始/结束触发、状态显示、日志和撤销规则。

单位和技能的定位文案分别写在 `UnitData.role_label` / `battle_note` 与 `SkillData.role_label` / `effect_note`。这些字段用于悬停说明和设计基线，不参与结算。技能说明应该描述机制事实，例如“范围 1”“移动-1”“自身下次行动提前 20%”“目标行动条-20%”，不要直接写“适合打 Boss/适合清小怪”这类攻略式提示。

临时 HUD、行动菜单、CTB 条、伤害预览、战前提示和结算面板都需要使用半透明深色背景、细边框和少量内边距。不要把关键文字直接裸放在战场格子上；新增临时 UI 时优先复用现有面板样式，避免每个区域各自发明背景。

战斗 UI 采用 v0 边缘布局：顶部低矮状态条放 CTB、同步率摘要和快捷按钮；底部指令栏放当前单位、操作按钮和悬停说明；右侧抽屉只在玩家打开日志或技能预览时展开。常驻 UI 不要占屏幕中央，日志、预览和未来单位详情需要复用右侧抽屉，避免多个面板同时覆盖棋盘。

棋盘单位悬停小卡只做轻量信息，不替代未来正式单位详情面板。它需要展示阵营、属性、HP/AP、速度、移动、技能概要和临时状态；当鼠标位于顶部 HUD、底部指令栏、右侧抽屉、战前/结算弹窗等 UI 区域时，小卡必须隐藏。

## 敌方 AI 约定
敌方 AI 目标选择采用轻量评分，而不是固定攻击训练师或只攻击最近单位。评分需要综合上一次攻击者、属性克制、低血/可斩杀目标、距离、支援单位、增益状态和敌人属性身份倾向；这样敌人行为仍可读，但不会每次都像固定脚本。

敌方战术偏好由 `UnitData.ai_profile` 指向 `Enums.AIProfile`。当前 profile 包括：`BALANCED` 通用、`HUNTER` 猎手/收割残血、`ELEMENTAL` 克制型、`GUARDIAN` 守卫/最近目标、`SUPPRESSOR` 压制训练师/支援/带增益单位、`AREA` 范围覆盖。新增敌人时优先选择或微调 profile，不要在 `UnitAI.run()` 中为单个怪写专用分支。

近战敌人优先逼近目标；远程敌人在已经进入射程时应倾向保持射程，不要无意义贴脸。厚血大怪的蓄力目标优先选择能覆盖更多我方单位的位置，同时仍要尊重目标评分，避免预警总是落在同一个训练师身上。

## 地形约定
地形数据由 `Grid.gd` 的 `_terrain[y][x]` 维护。高亮清除时必须恢复地形颜色，而不是统一恢复普通地面颜色。

当前 MVP 只启用普通地面和草地显示，不启用燃烧地面、点燃草地或地形伤害。若后续恢复地形伤害，需要重新检查移动撤回规则。

## 敌方预警
厚血大怪的蓄力攻击使用 `Grid.gd` 的 warning cells 显示橙色预警格。普通移动/攻击范围高亮可以临时覆盖预警色，但清除高亮后必须恢复预警色。大怪死亡或蓄力结算时需要清除预警格。
