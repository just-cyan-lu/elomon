# elomon

`elomon` 是一个使用 Godot 4.6 开发的宝可梦风格战棋 RPG MVP 原型。当前版本的目标不是做完整游戏，而是验证一场战斗中的核心循环：CTB 行动顺序、网格移动、属性克制、训练师同步率操作、后备能力提取、召唤/回收、伤害预览、行动日志和轻量状态系统。

## 当前范围

- 引擎：Godot 4.6
- 主场景：`battle/Battle.tscn`
- 视口：`640x360`
- 战场：`20x20` 网格，单格 `32px`
- 战斗模型：CTB，单位按速度积累 AP，达到 100 AP 时行动
- 内容状态：MVP 样板战，单位和技能数据暂时由脚本动态生成

`addons/godot_mcp` 是 Godot 编辑器自动化插件，不属于游戏逻辑。

## 运行方式

用 Godot 4.6 打开项目，按 F5 运行。

项目当前从 `battle/Battle.tscn` 启动。由于战场尺寸是 `640x640`，而当前视口是 `640x360`，完整战场会超出首屏。

常用 headless 解析检查：

```bash
/Applications/Godot.app/Contents/MacOS/Godot --headless --path . --quit
```

## 战斗流程

1. `Battle.gd` 初始化战斗场景，并显示战前准备面板。
2. 玩家选择样板战、2 只开局宝可梦和训练师默认提取形态。
3. `Battle.gd` 动态创建 MVP 所需的 `UnitData`、`SkillData`、`SkillEffectData`。
4. 战前提示面板说明胜利目标、属性克制、同步率限制、指令卡和敌方范围按钮。
5. 玩家点“开始行动”后，`CTBSystem.gd` 开始推进 AP。
6. 单位 AP 达到 100 后，CTB 暂停，`Battle.gd` 进入玩家回合或敌方回合。
7. 玩家回合通过 `ActionMenu.gd` 选择移动、技能、同步率操作、回收、待机或结束行动。
8. 敌方回合由 `UnitAI.gd` 结算，AI 返回结构化日志给 `Battle.gd` 展示。
9. 回合结束时扣除行动 AP，清理临时选择和高亮，更新敌方威胁范围，然后恢复 CTB。
10. 任一阵营无存活单位时结算战斗，结算面板显示击败数、同步率操作数、训练师状态和剩余同步率。

## 战斗系统

### CTB 与 AP

每个单位按速度持续获得 AP。AP 达到 100 后，该单位进入可行动状态，CTB 暂停。

当前 AP 是内部行动节奏值：

- 标准技能通常消耗 100 AP。
- 快招消耗较少，UI 表述为“自身下次行动提前”。
- 重招消耗较多，UI 表述为“自身下次行动推后”。
- 未使用技能时默认消耗 50 AP，即更快进入下一次行动。
- 移动不消耗 AP。

玩家文案尽量描述行动顺序影响，而不是让玩家心算移动和技能共用 AP。

### 网格行动

每个单位有固定移动距离，来自 `UnitData.move_range`。`Grid.gd` 使用曼哈顿距离计算移动范围和攻击范围。

玩家技能流程：

1. 选择技能。
2. 点击目标格进入预览。
3. 再次确认同一目标后结算。
4. 范围技能以目标格为中心计算影响单位。

如果玩家移动后还没有结算不可逆行动，可以按 Esc 撤回到本回合开始状态。

### 属性克制

当前 MVP 使用 7 个属性：

- 火
- 水
- 草
- 冰
- 雷
- 飞
- 地

属性倍率由 `TypeChart.gd` 处理。单属性克制为 `2x`，抵抗为 `0.5x`。单位属性以列表形式保存，因此未来双属性可以自然连乘得到 `4x` 或 `0.25x` 等结果。

### 技能与附加效果

`SkillData` 保存技能的基础事实：

- 技能名
- 定位标签
- 伤害或治疗值
- 射程
- 行动 AP 结算值
- 属性
- 范围半径
- 伤害/治疗类型
- 附加效果列表

附加效果放在 `SkillData.effects` 中，每项为 `SkillEffectData`。

当前已接入的效果类型：

- `ADD_STATUS`：给目标添加状态，例如移动压制、防御变化。
- `AP_DELTA`：改变目标当前 AP，例如目标行动条后退 20%。

`move_penalty` 和 `target_ap_delay` 仍作为旧字段兼容层保留；新增技能效果应优先使用 `SkillEffectData`。

### 状态系统

状态系统目前保持小规模实现，但数据形状按后续扩展设计。

`StatusTypes.gd` 定义：

- 状态 id
- 状态/印记分类
- 正面/负面/中性
- 是否可驱散
- 触发时机
- 持续类型
- 叠加规则
- 短标签和说明

`StatusInstance.gd` 保存单位身上的运行时状态：

- 数值
- 层数
- 剩余触发次数
- 来源单位
- 持续和叠加元数据

`StatusResolver.gd` 在以下钩子触发状态：

- 行动开始
- 行动结束
- 使用技能
- 造成伤害前后
- 受到伤害前后
- 移动后

当前真正接入的 `StatusInstance` 示例：

- `DEFENSE_MOD`：在目标下次受伤前改变有效防御，然后在该次受伤中消耗。

移动压制目前仍使用 `Unit.gd` 中的旧轻量字段，以保持兼容。

### 同步率

同步率是训练师的战斗资源。

当前规则：

- 初始同步率为 100。
- 没有硬上限。
- 自然回复只补到 100。
- 主动收益可以超过 100。
- 战斗结束后不继承到下一场。
- 训练师攻击和宝可梦攻击可以获得同步率。

每个训练师回合最多结算 1 次同步率操作：

- 指令卡
- 提取
- 召唤

移动、技能和回收不占用这次同步率操作。

### 指令卡

当前 MVP 不做随机抽牌，使用固定指令栏。

已有指令卡：

- 高速组件：目标宝可梦下一次移动 +2。
- 小型护盾：目标友方获得护盾。
- 火力插件：目标友方下一次攻击伤害提高。
- 弱点标记：目标敌方下一次受到伤害提高。
- 战术换位：训练师和目标己方宝可梦交换位置。
- 属性校准：目标下一次伤害技能使用训练师当前提取属性结算。

### 提取、召唤和回收

训练师可以提取后备宝可梦能力。提取会改变训练师属性，并把训练师技能替换为配置的提取技能。

关键规则：

- 战前默认提取免费生效。
- 战斗中再次提取消耗 100 同步率。
- 当前提取形态会占用对应后备宝可梦。
- 被占用的后备宝可梦不能被召唤，直到训练师切换到其他提取形态。
- 额外召唤费用递增：100、160、220。
- 回收返还 50 同步率。
- 被回收的宝可梦保留当前 HP，再次召唤不会满血。
- 训练师回收不占用同步率操作次数。
- 宝可梦可以在自己的行动中回收自身，并结束该行动。

## 当前样板战

战前准备面板可以选择样板战。

- 第一场战斗：基础验证战，包含草属性厚血 Boss 和混合属性敌人。
- 第二样板战：侧翼压制测试，中线地属性压力，上下翼水/飞/草敌人制造站位压力。

这些关卡是临时试验台。当前关卡数据仍在 `Battle.gd` 中动态生成；等战斗形态稳定后，应迁移到数据资源。

## 日志与预览

战斗日志是结构化 record，包含展示文本和 metadata。日志记录已结算的移动、技能、指令卡、提取、召唤、回收、敌方行动、待机、AP 变化、同步率变化、伤害、目标信息和位置。

预览不会进入正式日志。可撤销的移动和训练师提取会先进入 pending logs，只有在玩家执行不可逆行动或结束行动后才提交。

技能预览通过 `action_preview_updated(preview)` 发出，包含行动者、目标格、AP 预估、同步率预估、目标 HP 变化、伤害、属性倍率和附加效果 metadata。

## 数据状态

为了快速迭代，当前大多数战斗数据仍在 `Battle.gd` 中动态创建：

- 玩家宝可梦
- 敌方单位
- 技能
- 提取定义
- 召唤定义
- 样板战敌人组合

仓库中已有的 `.tres` 资源属于旧原型数据，不是当前 MVP 战斗的主要数据源。

## 备注

- MVP 暂不做战斗内捕捉。
- 稳定度 / Break 字段仍保留为未来设计预留，但当前不展示也不结算。
- 单个技能仍按单属性结算；同一只宝可梦可以携带不同属性技能。
- 当前 UI 是用于试玩验证的临时方案，不是最终表现。
